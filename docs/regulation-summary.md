# Регламент FIRA Autonomous Cars (Physical Division) — сводка

Это пересказ своими словами, не дословная копия (полный текст правил защищён
авторским правом FIRA). Перед соревнованием обязательно свериться с
оригиналом:

- Официальные правила Pro: https://docs.google.com/document/d/1PgeKrsCEL-KnZFci-iQUgFKoVnY7qiXql9oOvQACfEY/
- Официальные правила Youth: https://docs.google.com/document/d/1pyhgvSQw7eaGDG0dzchA0VkbOYnGsd1_AVNhvs1iz8c/
- Страница лиги: https://firaworldcup.org/leagues/fira-challenges/autonomous-cars/

Версия документа на момент выжимки: **Laws of the Game (Pro) v2.0.1, обновлено 18 июня 2025**.

## Ключевые ограничения по машине

Уточнено (июль 2025 revision документа Pro): регламент нормирует не только
длину/высоту корпуса, но и **колею** (track width) и **колёсную базу**
(wheelbase) — это отдельные параметры, которые bounding box одной детали
STL не показывает напрямую.

| Параметр | Требование (Pro) | Наша модель (NITROUS, `models/`) |
|---|---|---|
| Масштаб | 1:10 относительно реального автомобиля | По заявлению автора — обычный любительский RC-масштаб |
| Максимальная длина | 600 мм | Корпус (`Print-01_Main-body`) — 187 мм по Y — с большим запасом |
| Максимальная ширина | 400 мм | Не измерено напрямую (см. предупреждение ниже) |
| Максимальная высота | 450 мм | Макс. высота детали (`Print-10_Central-Axis`) — 61 мм — с запасом |
| **Колея (track width)** | **150–350 мм** | **⚠️ не измерено — см. предупреждение ниже** |
| **Колёсная база (wheelbase)** | **200–550 мм** | **⚠️ не измерено — см. предупреждение ниже** |
| Рулевое управление | Ackermann, минимум одна управляемая ось | Модель спроектирована с Ackermann-геометрией (`Print-09_Stearing`, `Print-11_Bridge`) — совпадает |
| Привод | Полный или передний | DC-мотор 280 + редуктор (`Print-02_Gearbox`) — тип привода уточнить по сборке |
| Питание | Электрическое, батареи без ограничений | DC-мотор + сервопривод MG90S (`docs/electronics-bom.md`) — совпадает |
| Готовые платформы (DonkeyCar, JetRacer и т.п.) | **Разрешены в Pro** (запрещены только в Youth) | Не используются пока, см. `EXTERNAL_SOURCES.md` про DonkeyCar |
| Датчики следования по линии | **ИК-датчики line-following явно запрещены** | Не актуально — используется камера, не ИК (см. `lane-detection-adaptation.md`) |

### ⚠️ Критическое предупреждение: колея и колёсная база не проверены

Bounding box отдельных STL-деталей (`tools/stl_bbox.py`) даёт размер
**каждой детали как она лежит в файле для печати**, а не расстояние между
осями в собранной машине — колёсную базу и колею по этим числам напрямую
не вычислить. Судя по компактному размеру корпуса (187 мм по самой длинной
стороне), **есть реальный риск, что колёсная база модели NITROUS меньше
минимума в 200 мм**, а колея — меньше минимума в 150 мм: это машина
скорее любительского "мелкого" RC-масштаба, а не полноразмерная 1:10
гоночная платформа, для которой калибровался этот регламент.

**Это нужно измерить руками при сборке шасси** (расстояние между
передней и задней осью, расстояние между центрами левого/правого колеса) —
до этого момента нельзя считать вопрос габаритов закрытым. Если колёсная
база/колея окажутся меньше минимума — потребуется либо доработать шасси
(удлинить раму между Print-05/Print-06 и осями), либо рассмотреть другую
модель/масштабирование STL при печати.

## Формат соревнования

- Два этапа: **предварительный** и **финальный**, при переходе в финал баллы
  обнуляются.
- Две дисциплины:
  1. **Race Track** — гонка по размеченной трассе на скорость/чекпоинты.
  2. **Urban Track** — городская трасса с дорожными знаками и препятствиями,
     часть баллов начисляется за точность распознавания знаков.
- На каждую трассу — несколько попыток с ограничением по времени и
  интервалом на подготовку между попытками (точные числа см. в оригинале
  документа — они могут меняться от года к году).
- На трассе Pro (в отличие от Youth) могут быть **препятствия** — как на
  Race Track, так и на Urban Track; столкновение с препятствием завершает
  попытку. В Youth-версии препятствий на трассе нет вообще — это одно из
  основных упрощений младшей категории.
- Безопасность: капитан команды может прервать попытку командой **STOP**.
  Если кто-то прикасается к машине без предварительной команды STOP —
  попытка аннулируется. Явных требований к аппаратному аварийному стопу
  (kill switch) в тексте не встретилось — стоит перепроверить перед
  соревнованием, обычно это негласный стандарт для подобных лиг.

## Датчики

- **Явно запрещены**: инфракрасные датчики следования по линии (ИК line
  sensors) — то есть классический подход "ИК-массив под днищем" для этого
  регламента не подходит в принципе, и наш выбор камеры + компьютерное
  зрение (см. `lane-detection-adaptation.md`) — не просто один из
  вариантов, а фактически обязательный подход по датчикам этого класса.
- Также запрещена **беспроводная связь во время попытки** (по Youth-версии;
  для Pro стоит перепроверить отдельно, т.к. в изученном фрагменте текста
  это не встретилось явно, но раз множитель баллов различает
  onboard/offboard обработку — предполагается, что вся логика работает
  автономно на борту без внешней связи по умолчанию).
- Лидар/ультразвук/GPS — ни разрешение, ни запрет не встретились в
  изученных фрагментах документа; трактовать как "не запрещено прямо",
  но лучше подтвердить у организаторов перед закупкой дорогого сенсора.

## Готовые платформы

Pro-версия регламента **разрешает** использование одного автомобиля/платформы
целиком (включая коммерческие: DonkeyCar, NVIDIA JetRacer и т.п.) — правило
"нельзя менять платформу между попытками, но можно чинить/дорабатывать
компоненты". Это отличается от Youth-версии, где такие платформы прямо
запрещены. Значит рекомендация из `EXTERNAL_SOURCES.md` (оценить DonkeyCar
как основу) — не противоречит регламенту, если проект остаётся в Pro.

## Автономность и множитель баллов

Регламент явно различает:
- **Onboard processing** (вся обработка данных — на самой машине) — полный
  множитель баллов;
- **Offboard processing** (данные уходят на внешний компьютер и обратно) —
  пониженный множитель.

Это прямое требование к архитектуре: конвейер lane detection / детекции
знаков (см. [`lane-detection-adaptation.md`](lane-detection-adaptation.md))
должен работать целиком на бортовом одноплатнике, а не стримиться на ноутбук
для обработки.
